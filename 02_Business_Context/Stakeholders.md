# Стейкхолдеры

## 1. Цель документа

Этот документ описывает ключевых стейкхолдеров проекта **Digital Bank Credit Decision Platform** — платформы обработки заявок на кредитные карты для условного цифрового банка NovaBank.

Документ нужен, чтобы понять:

- кто участвует в процессе обработки кредитной заявки;
- какие цели и интересы есть у каждой стороны;
- какие боли решает система;
- кто влияет на требования к продукту;
- кто будет использовать систему напрямую или косвенно.

## 3. Описание стейкхолдеров

### 3.1 Клиент

**Описание:**  
Клиент — физическое лицо, которое хочет оформить кредитную карту через сайт или мобильное приложение NovaBank.

**Цели клиента:**

- быстро подать заявку;
- получить понятное решение;
- отслеживать статус заявки;
- получить оффер с лимитом, ставкой и условиями;
- не вводить лишние данные повторно.

**Боли клиента:**

- долгое ожидание решения;
- непонятный статус заявки;
- отказ без объяснения;
- необходимость повторно предоставлять данные;
- отсутствие уведомлений.

**Что важно для требований:**

- простая форма заявки;
- понятные статусы;
- уведомления об изменениях;
- минимальное количество ручных действий;
- защита персональных данных.

---

### 3.2 Риск-аналитик

**Описание:**  
Риск-аналитик рассматривает заявки, которые система не может одобрить или отклонить автоматически. Обычно это заявки со средним уровнем риска или спорными признаками.

**Цели риск-аналитика:**

- видеть заявки, требующие ручной проверки;
- понимать причину попадания заявки на manual review;
- видеть результаты KYC, скоринга и антифрода;
- принимать решение: одобрить или отклонить;
- фиксировать причину решения.

**Боли риск-аналитика:**

- недостаточно данных для принятия решения;
- непонятная причина ручной проверки;
- нет истории предыдущих решений;
- приходится переключаться между несколькими системами;
- сложно отследить SLA обработки.

**Что важно для требований:**

- очередь заявок на ручную проверку;
- отображение score и risk level;
- отображение результатов KYC и антифрода;
- возможность принять решение вручную;
- обязательное указание причины решения;
- audit log всех действий.

---

### 3.3 Сотрудник поддержки

**Описание:**  
Сотрудник поддержки помогает клиентам, которые обращаются с вопросами по статусу заявки.

**Цели сотрудника поддержки:**

- быстро найти заявку клиента;
- увидеть текущий статус;
- объяснить клиенту, на каком этапе находится заявка;
- не иметь доступа к лишним персональным данным;
- не влиять на кредитное решение.

**Боли сотрудника поддержки:**

- клиент спрашивает статус, а информация недоступна;
- статусы непонятны;
- нет истории изменений;
- слишком много чувствительных данных в интерфейсе;
- невозможно понять, куда передать проблему.

**Что важно для требований:**

- поиск заявки по телефону, email или application_id;
- отображение текущего статуса;
- отображение краткой истории статусов;
- маскирование паспортных данных;
- запрет на изменение кредитного решения;
- понятные клиентские формулировки статусов.

---

### 3.4 Кредитный менеджер

**Описание:**  
Кредитный менеджер отвечает за эффективность кредитного продукта и показатели воронки заявок.

**Цели кредитного менеджера:**

- понимать, сколько заявок поступает;
- видеть долю одобрений и отказов;
- анализировать причины отказов;
- видеть долю manual review;
- оценивать эффективность автоматизации;
- отслеживать конверсию в принятие оффера.

**Боли кредитного менеджера:**

- данные разбросаны по разным системам;
- сложно понять, где клиенты выпадают из процесса;
- нет единого dashboard;
- нет точных метрик по времени обработки;
- manual review может занимать слишком много времени.

**Что важно для требований:**

- метрики по кредитной воронке;
- отчёты по статусам заявок;
- аналитика причин отказов;
- аналитика времени обработки;
- dashboard по ключевым KPI;
- выгрузка данных для анализа.

---

### 3.5 Служба безопасности

**Описание:**  
Служба безопасности анализирует подозрительные заявки и результаты антифрод-проверок.

**Цели службы безопасности:**

- выявлять мошеннические заявки;
- видеть признаки подозрительной активности;
- анализировать результаты антифрод-проверки;
- блокировать подозрительные заявки;
- передавать спорные случаи на дополнительную проверку.

**Боли службы безопасности:**

- не видно причин антифрод-срабатывания;
- нет истории действий по заявке;
- сложно отличить ошибочное срабатывание от реального риска;
- недостаточно данных для расследования;
- подозрительные заявки могут пройти автоматически.

**Что важно для требований:**

- отдельный результат антифрод-проверки;
- fraud indicators;
- возможность отправить заявку на дополнительную проверку;
- логирование всех действий;
- сохранение истории проверок;
- ограничение доступа к чувствительным данным.

---

### 3.6 Compliance / юридический отдел

**Описание:**  
Compliance и юридический отдел отвечают за соблюдение требований к обработке данных, проверкам клиента и прозрачности процесса.

**Цели compliance / юридического отдела:**

- обеспечить корректную обработку персональных данных;
- контролировать доступ к чувствительной информации;
- обеспечить хранение истории действий;
- снизить юридические риски;
- убедиться, что клиент получает корректные уведомления.

**Боли compliance / юридического отдела:**

- персональные данные доступны слишком широкому кругу сотрудников;
- отсутствует audit log;
- непонятно, кто и когда изменил решение;
- нет единых правил хранения истории;
- клиенту могут показываться некорректные причины отказа.

**Что важно для требований:**

- ролевая модель доступа;
- маскирование персональных данных;
- audit log;
- хранение истории статусов;
- контроль доступа к данным;
- корректные уведомления клиенту.

---

### 3.7 IT-команда

**Описание:**  
IT-команда отвечает за реализацию системы: backend, frontend, интеграции, база данных, тестирование и инфраструктура.

**Цели IT-команды:**

- получить понятные требования;
- понимать бизнес-логику;
- видеть статусы и переходы;
- иметь API-контракты;
- понимать структуру данных;
- знать edge cases и ошибки внешних сервисов.

**Боли IT-команды:**

- требования неполные или противоречивые;
- нет единого списка статусов;
- не описаны альтернативные сценарии;
- неясно, что делать при ошибках внешних сервисов;
- API-контракты не согласованы;
- бизнес-правила описаны неформально.

**Что важно для требований:**

- функциональные требования;
- нефункциональные требования;
- business rules;
- state model;
- sequence diagram;
- ERD;
- API specification;
- JSON examples;
- acceptance criteria.

---

### 3.8 Системный администратор

**Описание:**  
Системный администратор управляет пользователями, ролями и доступами внутри платформы.

**Цели системного администратора:**

- создавать и отключать пользователей;
- назначать роли;
- управлять правами доступа;
- контролировать безопасность системы;
- видеть технические события и ошибки.

**Боли системного администратора:**

- роли не разграничены;
- пользователи имеют лишние права;
- сложно понять, кто получил доступ к данным;
- нет журналирования действий;
- нет понятной матрицы доступов.

**Что важно для требований:**

- ролевая модель;
- матрица доступов;
- audit log;
- управление пользователями;
- ограничение доступа к персональным данным;
- журналирование изменений прав.

---

### 3.9 Руководство банка

**Описание:**  
Руководство банка является спонсором проекта и заинтересовано в бизнес-эффекте от автоматизации.

**Цели руководства:**

- сократить время обработки заявок;
- снизить операционные расходы;
- повысить конверсию клиентов;
- уменьшить долю ручной обработки;
- повысить прозрачность процесса;
- получить управленческую отчётность.

**Боли руководства:**

- процесс обработки заявок непрозрачен;
- сложно оценить эффективность сотрудников и автоматизации;
- высокая доля ручной работы;
- клиенты уходят из-за долгого ожидания;
- нет единой картины по воронке заявок.

**Что важно для требований:**

- KPI по заявкам;
- dashboard для руководства;
- метрики по времени обработки;
- approval rate;
- rejection rate;
- manual review rate;
- offer acceptance rate.

## 4. RACI-матрица

RACI-матрица показывает распределение ответственности по ключевым процессам.

Обозначения:

- **R** — Responsible, выполняет действие;
- **A** — Accountable, несёт ответственность за результат;
- **C** — Consulted, консультирует;
- **I** — Informed, получает информацию.

| Процесс / Роль | Клиент | Риск-аналитик | Поддержка | Кредитный менеджер | Служба безопасности | Compliance | IT-команда | Администратор |
|---|---|---|---|---|---|---|---|---|
| Подача заявки | R | I | I | I | I | I | C | I |
| Валидация данных | I | I | I | C | I | C | R | I |
| KYC-проверка | I | C | I | I | I | C | R | I |
| Скоринговая проверка | I | C | I | A | I | C | R | I |
| Антифрод-проверка | I | C | I | I | A/R | C | R | I |
| Ручная проверка | I | A/R | I | C | C | I | C | I |
| Формирование оффера | I | C | I | A | I | C | R | I |
| Уведомление клиента | I | I | C | A | I | C | R | I |
| Настройка ролей | I | I | I | C | C | C | C | A/R |
| Анализ метрик | I | C | I | A/R | C | I | C | I |

## 6. Вывод

В проекте есть несколько групп стейкхолдеров с разными интересами:

- клиенту важна скорость, понятность и удобство;
- бизнесу важны конверсия, эффективность и контроль;
- риск-аналитикам важна полнота данных для принятия решений;
- службе безопасности важны признаки мошенничества и история проверок;
- compliance важны доступы, audit log и защита персональных данных;
- IT-команде важны точные требования, статусы, API и модель данных.

Поэтому проект должен учитывать не только клиентский путь, но и внутренние процессы банка: ручную проверку, контроль рисков, аудит действий, разграничение доступов и аналитику по кредитной воронке.
